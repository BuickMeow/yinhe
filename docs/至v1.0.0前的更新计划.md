## 你的计划可能漏了什么

1. **性能基准测试** — 你说要支撑1亿音符，但计划里没有任何性能测试/优化项。至少要有一个"加载1000万音符工程测试"或"极端缩放卡顿优化"

2. **撤销系统的完整性** — 目前只提了一.5里的撤回问题。undo/redo 是整个编辑器的基石，应该系统地验证每个操作是否都能正确撤销

3. **MIDI 导出/导入的往返测试** — 我们刚修了导出 bug，但应该建立系统性的测试用例来保证各种 MIDI 数据（RPN/NRPN/Sysex/Meta/中文轨道名）都能无损往返

4. **音频设备切换处理** — 你已经在四.8提了，但这是个大坑。插拔耳机、蓝牙断开、默认设备变更都会导致音频引擎崩溃。建议提一个专门的调研项

5. **量化功能本身** — 你只提了"两套量化"的存储问题，但**量化功能本身**（量化音符到网格、量化长度、人性化等）没出现在计划里。这是 MIDI 编辑器的核心功能

6. **文件关联 / 系统集成** — `.yin` 文件双击打开、macOS 上设为默认打开方式、最近文件列表。这些在 v1.0 前需要搞定

7. **首选项持久化** — 快捷键配置、默认 SF 路径、音频设备选择、界面缩放等设置是否保存和恢复

8. **Loop/范围播放** — 走带控制的基本功能，没在计划里

9. **音符选择工具** — 框选、加选(Shift)、范围选、全选(Ctrl+A)的交互一致性

10. **音高/力度预览** — 钢琴卷帘上点按音符时是否发声预览

建议加一个 **P4 — v1.0后考虑** 区，把这些放进去，先不做但留个记录。

## 1. 框选拖动会把没选中的音符带走（旧问题，找到代码根源了）

之前跟你提过的那个框选 bug，这次翻了下代码，根源确认：

- `crates/yinhe-editor-core/src/batch_ops.rs:25` 的 `remove_selected` 是按**调用那一刻**的 selection rect 现查音符的；
- `crates/yinhe-editor-core/src/document.rs:826` 每帧移动完立刻 `selection.offset(...)` 让框跟着走。

也就是说拖动过程中每一帧都拿"框当前盖住的东西"重新算移动集，框扫过哪里，哪里的音符就被卷进来——不是只有最初选中的那批在动。

修法方向：拖动开始时把命中音符的身份快照一次（note id 或 `(track, start_tick, dup_index)` 列表），拖动期间只动快照里的，别按框现查。松手再统一提交 undo。

## 3. device_lost 没有恢复路径 ✅ 已修（最小版）

`yinhe-egui/src/render_context.rs:84` 设了 device-lost 回调置 flag，然后 `:212` / `:305` 检测到 flag 就**永久跳过所有 GPU 渲染**，没有任何重建 device 或提示用户的逻辑。

Windows 上显卡驱动 TDR（重负载下不算罕见）触发一次 device lost，你的钢琴卷帘就永久停更了，用户只能自己悟出要重启。建议至少弹个对话框提示重启，正经做法是走 eframe/wgpu 的图形上下文重建。

**已实施**：选了 doc 建议的"至少弹个对话框提示重启"路径：
- `RenderContext::device_lost()` 暴露 flag 给 UI
- 新增 `crates/yinhe-egui/src/dialogs/gpu_device_lost.rs`，标题"需要重启"，按"退出 yinhe"会触发 `should_exit`
- 在 `app/dialog_dispatch.rs::show_dialogs` 每帧 OR 检测 `render_ctx.device_lost() || arr_render_ctx.device_lost() || audio_state.handle.handle.stream_error()`，任一为 true 即弹窗
- cpal 流错误也走同一对话框（renderer 死了同样无救）

**遗留问题**（顺手记下，以后再翻）：
- `wgpu::Device::set_device_lost_callback` 会替换前一个回调，`render_ctx` 先注册、`arr_render_ctx` 后注册，最终只有 `arr_render_ctx` 的回调会触发——`render_ctx.device_lost` 实际是死状态。当前用 OR 兜底，但根因是没共享 `Arc<AtomicBool>`。要让 `RenderContext::new` / `from_render_state` 接受外部传入的 `Arc<AtomicBool>` 才能彻底修。
- 真正的恢复路径（eframe/wgpu 图形上下文重建）还没做，目前用户只能退出重启。
